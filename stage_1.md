#### Шаг 1.1. Сбор и классификация негативных ситуаций

**Цель:** создать исчерпывающую базу признаков для детекции негативных ситуаций.


**1. Словарь ключевых фраз:**

* **Угрозы жалобы:**
    * «подам жалобу»;
    * «обращусь в суд»;
    * «напишу в Роспотребнадзор»;
    * «сообщу в вышестоящую организацию»;
    * «опубликую отзыв в интернете».
* **Нецензурная лексика:**
    * мат (базовый список из 10–15 основных слов);
    * оскорбления («некомпетентный», «бестолковый», «идиот»);
    * унижения («вы ничего не понимаете», «вы бесполезны»).
* **Признаки конфликта:**
    * повышение тона («я не буду это терпеть!»);
    * перебивание («дайте мне договорить!»);
    * категоричные формулировки («никогда», «ни за что»).
* **Негативные эмоции:**
    * разочарование («опять та же проблема»);
    * раздражение («сколько можно объяснять?»);
    * гнев («вы меня злите!»).

**2. Акустические признаки:**

| Признак | Порог | Метод измерения |
|--------|-------|--------------|
| Громкость | > 70 дБ | RMS‑анализ |
| Тон голоса | > 250 Гц (для женщин), > 180 Гц (для мужчин) | FFT‑анализ |
| Паузы между репликами | < 0.5 с (перебивание) | Анализ тишины |
| Скорость речи | > 200 слов/мин | WPM‑расчёт |
| Перекрывающиеся реплики | > 3 с одновременной речи | VAD + сравнение каналов |


**3. Шкала серьёзности ситуаций:**

| Уровень | Тип ситуации | Вес | Действия |
|--------|------------|-----|----------|
| Критический (9–10) | Угроза немедленной жалобы, нецензурная брань | 10 | Мгновенное оповещение супервизора |
| Высокий (7–8) | Явный конфликт, сильное раздражение | 7 | Оповещение в течение 1 мин |
| Средний (4–6) | Лёгкое недовольство, намёк на жалобу | 4 | Фиксация в отчёте |
| Низкий (1–3) | Нейтральный тон с единичными негативными словами | 2 | Мониторинг без оповещения |

#### Шаг 1.2. Выбор моделей для детекции

**1. Модели распознавания речи (ASR):**

* **Yandex SpeechKit** — высокая точность для русского языка, поддержка потоковой обработки.
* **Google Cloud Speech‑to‑Text** — мультиязычная модель, хорошая обработка акцентов.
* **Vosk** — оффлайн‑решение для локальной установки.

**Критерии выбора:**
* точность ≥ 90 % на русском языке;
* задержка ≤ 3 с;
* поддержка потокового режима.

**2. Модели классификации эмоций:**

* **RuBERT‑Emotion** — дообученная модель для русского языка.
* **DeepPavlov Emotion** — специализированное решение для эмоций в речи.
* **Custom CNN** — свёрточная сеть на основе спектрограмм.

**Классы эмоций:**
* гнев;
* раздражение;
* разочарование;
* нейтральное;
* положительное.

**3. Кастомные правила:**

* регулярные выражения для ключевых фраз;
* словари стоп‑слов;
* шаблоны конфликтных диалогов.

#### Шаг 1.3. Проектирование архитектуры

**Схема потоков данных:**

```
Аудиопоток → Предобработка → Транскрибация → Анализ текста + Анализ аудио → Детекция ситуаций → Оповещение → Хранение
```

**Компоненты системы:**

1. **Входной модуль:**
   * приём аудиопотоков (RTP, WebRTC);
   * нормализация громкости;
   * разделение каналов (оператор/клиент).
2. **Ядро анализа:**
   * ASR‑движок;
   * детектор эмоций;
   * анализатор ключевых слов;
   * акустический анализатор.
3. **Механизм принятия решений:**
   * агрегация сигналов;
   * расчёт уровня негатива;
   * классификация ситуации.
4. **Модуль оповещений:**
   * Telegram‑бот;
   * Slack‑вэбхук;
   * email‑отправка;
   * интеграция с Битрикс24.
5. **Хранилище данных:**
   * PostgreSQL для событий;
   * MinIO для аудиофрагментов;
   * Redis для кэша.
6. **Дашборд:**
   * метрики в реальном времени;
   * история событий;
   * фильтры по операторам/клиентам.

**API‑эндпоинты:**

* `POST /api/v1/analyze` — старт анализа звонка;
* `GET /api/v1/events` — получение событий;
* `PUT /api/v1/rules` — обновление правил детекции.

### Шаг 1.4. Разработка стратегии оповещений

#### 1. Приоритезация по типам ситуаций

Создаём систему приоритетов на основе серьёзности и срочности:

| Тип ситуации | Описание | Приоритет | Действие |
|-----------|---------|---------|---------|
| КРИТИЧЕСКАЯ | Прямая угроза жалобы, нецензурная лексика в адрес оператора, явный конфликт | Высокий (1) | Мгновенное оповещение супервизора, запись фрагмента |
| СЕРЬЁЗНАЯ | Признаки раздражения, недовольство клиента, намёк на жалобу | Средний (2) | Оповещение в течение 5 минут, запись фрагмента |
| ПОТЕНЦИАЛЬНАЯ | Лёгкое недовольство, затянувшаяся пауза, снижение темпа речи | Низкий (3) | Ежедневный отчёт, запись при повторении |
| МОНИТОРИНГ | Нейтральные сигналы, требующие наблюдения | Очень низкий (4) | Еженедельный отчёт |

#### 2. Каналы доставки оповещений

**Поддерживаемые каналы:**
* Telegram (бот с кнопками действий);
* Slack (вэбхук с интерактивными элементами);
* Email (структурированные письма);
* Битрикс24 (автоматическое создание задач);
* Веб‑интерфейс колл‑центра (всплывающие уведомления).

**Матрица каналов по приоритетам:**

| Приоритет | Telegram | Slack | Email | Битрикс24 | Веб‑интерфейс |
|---------|---------|-------|-------|----------|-------------|
| Высокий | ✅ (с звуком) | ✅ (с меткой @here) | ✅ | ✅ (срочная задача) | ✅ (всплывающее окно) |
| Средний | ✅ | ✅ | ✅ | ✅ (обычная задача) | ✅ |
| Низкий | — | — | ✅ (в дайджесте) | — | ✅ (в ленте событий) |
| Очень низкий | — | — | ✅ (еженедельный отчёт) | — | — |

#### 3. Расписание уведомлений

**Режимы работы:**
* **Рабочие часы** (9:00–18:00): все оповещения в полном объёме.
* **Внерабочее время**:
   * высокий приоритет — все каналы;
   * средний приоритет — только Telegram/Slack;
   * низкий и очень низкий — откладываются до утра.
* **Экстренный режим**: все оповещения независимо от времени (активируется вручную).

#### 5. Система эскалации

**Правила эскалации:**
1. Если супервизор не подтвердил оповещение в течение 2 минут (высокий приоритет), уведомление отправляется старшему супервизору.
2. При повторении аналогичных ситуаций у одного оператора ≥ 3 раз за неделю — автоматическое уведомление руководителю отдела.
3. При обнаружении системной проблемы (например, 5+ жалоб на один продукт) — оповещение product‑менеджера.

#### 6. Настройки персонализации

**Кастомизация для пользователей:**
* выбор предпочтительных каналов;
* настройка расписания (индивидуальные рабочие часы);
* фильтры по операторам (только свои подчинённые);
* пороги чувствительности (строгий/умеренный/расслабленный режим);
* шаблоны сообщений (возможность редактирования).

#### 7. Механизм подтверждения и обратной связи

**Статусы оповещений:**
* `new` — получено, ожидает обработки;
* `acknowledged` — подтверждено супервизором;
* `resolved` — ситуация разрешена;
* `escalated` — передана на следующий уровень.

**Действия в интерфейсе:**
* «Подтвердить» — отметить как просмотренное;
* «Взять в работу» — назначить себя ответственным;
* «Эскалировать» — передать старшему;
* «Отложить» — перенести на позже (с указанием причины);
* «Ложное срабатывание» — сообщить о ошибке системы.

#### 8. Логирование и аудит оповещений

**Поля лога:**
* `event_id` — идентификатор события;
* `notification_id` — идентификатор уведомления;
* `channel` — канал отправки;
* `sent_at` — время отправки;
* `read_at` — время прочтения;
* `action_taken` — принятое действие;
* `response_time` — время реакции;
* `user_id` — ID пользователя, получившего уведомление.

#### 9. Тестирование стратегии

**Чек‑лист тестирования:**
1. Проверка отправки по всем каналам для каждого приоритета.
2. Проверка расписания (рабочие/внерабочее время).
3. Проверка эскалации (тайм‑ауты, передача уровней).
4. Проверка персонализации (индивидуальные настройки).
5. Проверка механизма подтверждения (смена статусов).
6. Нагрузочное тестирование (100+ одновременных оповещений).
7. Проверка обработки ошибок (недоступность канала).

#### 10. Документация и обучение


**Материалы:**
* руководство по настройке оповещений;
