### Задание: Система выявления негативных ситуаций

**Цель:** создать сервис реального времени для анализа разговоров и оповещения супервизоров о негативных ситуациях (конфликт, угроза жалобы, нецензурная лексика) с использованием моделей детекции эмоций и ключевых слов на русском языке.


#### Требования к системе

**Функциональные требования:**
1. **Анализ в реальном времени:** обработка аудиопотока с задержкой ≤ 5 секунд.
2. **Детекция негативных ситуаций:**
   * конфликтные ситуации (повышение тона, перебивание);
   * угрозы жалобы («подам жалобу», «обращусь в суд»);
   * нецензурная лексика (мат, оскорбления);
   * негативные эмоции (гнев, раздражение, разочарование);
   * признаки агрессии (резкие интонации, повышенные децибелы).
3. **Оповещение супервизоров:**
   * мгновенные уведомления в Telegram/Slack;
   * создание задачи в Битрикс24;
   * подсветка в интерфейсе колл‑центра.
4. **Запись критических моментов:** автоматическое сохранение аудиофрагментов (10–30 секунд до/после события).
5. **Дашборд мониторинга:** визуализация статистики по негативным ситуациям.
6. **API для интеграции:** предоставление данных другим системам.
7. **История событий:** хранение логов ≥ 90 дней.

**Нефункциональные требования:**
* точность детекции ≥ 85 %;
* задержка оповещения ≤ 10 секунд;
* поддержка ≥ 50 одновременных звонков;
* аптайм ≥ 99,9 %;
* потребление памяти ≤ 500 МБ на поток;
* совместимость с основными форматами аудио (MP3, WAV, OPUS);
* масштабируемость (добавление новых правил детекции без остановки сервиса).

#### Критерии приёмки
1. Система корректно выявляет ≥ 85 % негативных ситуаций на тестовой выборке из 200 разговоров.
2. Оповещения приходят супервизору в течение 10 секунд после события.
3. Аудиофрагменты сохраняются с точностью ±2 секунды от события.
4. Дашборд отображает:
   * текущую нагрузку (звонки в обработке);
   * статистику по типам негативных ситуаций;
   * динамику по операторам;
   * карту горячих точек (частые проблемы).
5. API возвращает данные в формате JSON с тайм‑кодами событий.
6. Логи содержат полную информацию о:
   * обработанных звонках;
   * выявленных ситуациях;
   * отправленных уведомлениях;
   * ошибках системы.

### Этапы реализации

#### Этап 1. Анализ и проектирование
1. Сбор и классификация негативных ситуаций:
   * составление словаря ключевых фраз (жалобы, угрозы, мат);
   * определение акустических признаков конфликта (тон, громкость, паузы);
   * разработка шкалы серьёзности ситуаций.
2. Выбор моделей для детекции:
   * модели распознавания речи (ASR);
   * модели классификации эмоций;
   * кастомные правила на основе регулярных выражений.
3. Проектирование архитектуры:
   * схема потоков данных (аудио → анализ → оповещение);
   * БД для хранения событий;
   * API‑эндпоинты;
   * интерфейс дашборда.
4. Разработка стратегии оповещений:
   * приоритеты по типам ситуаций;
   * каналы доставки (Telegram, Slack, email, Битрикс24);
   * расписание уведомлений (рабочие часы/круглосуточно).

#### Этап 2. Подготовка окружения
1. Создание виртуального окружения Python.
2. Установка зависимостей:
   * библиотеки для обработки аудио;
   * модели ASR и эмоциональной классификации;
   * FastAPI/Flask для API;
   * SQLAlchemy для БД;
   * библиотеки интеграции с мессенджерами и Битрикс24.
3. Настройка PostgreSQL для хранения:
   * логов событий;
   * аудиофрагментов;
   * настроек оповещений.
4. Настройка системы логирования и мониторинга.

#### Этап 3. Разработка ядра анализа
1. Реализация модуля загрузки и предварительной обработки аудио:
   * нормализация громкости;
   * удаление шумов;
   * разделение на каналы (оператор/клиент).
2. Создание конвейера анализа:
   * транскрибация аудио в текст;
   * детекция эмоций в речи;
   * поиск ключевых фраз;
   * анализ акустических параметров (тон, темп, паузы).
3. Разработка системы правил:
   * регулярные выражения для ключевых слов;
   * пороговые значения для акустических параметров;
   * весовые коэффициенты для разных признаков.
4. Реализация механизма принятия решений:
   * агрегация сигналов от разных детекторов;
   * расчёт общего уровня негатива;
   * классификация ситуации по типам.
5. Создание модуля сохранения критических фрагментов:
   * буферизация аудио;
   * запись по триггеру;
   * сжатие и сохранение в хранилище.

#### Этап 4. Интеграция с каналами оповещения
1. Разработка коннекторов для:
   * Telegram (бот);
   * Slack (вэбхук);
   * email (SMTP);
   * Битрикс24 (REST API).
2. Реализация системы приоритетов:
   * критические ситуации (мгновенное оповещение);
   * средние (в течение 5 минут);
   * низкие (ежедневный отчёт).
3. Создание шаблонов уведомлений:
   * для разных каналов;
   * с контекстной информацией (оператор, клиент, тайм‑код).

#### Этап 5. Разработка дашборда
1. Проектирование UI:
   * главная панель с метриками;
   * карта негативных ситуаций;
   * детализация по операторам и клиентам;
   * просмотр аудиозаписей.
2. Реализация фильтров:
   * по типам ситуаций;
   * по операторам;
   * по периодам;
   * по серьёзности.
3. Создание визуализаций:
   * графики динамики;
   * тепловые карты;
   * круговые диаграммы распределения.
4. Разработка модуля отчётов:
   * ежедневные/еженедельные PDF/Excel;
   * экспорт данных для BI‑систем.

#### Этап 6. Тестирование
1. Модульное тестирование:
   * модуля обработки аудио;
   * детектора эмоций;
   * системы правил;
   * коннекторов оповещений.
2. Интеграционное тестирование:
   * сквозной сценарий (аудио → оповещение);
   * нагрузка (50+ одновременных потоков);
   * отказоустойчивость (обрыв соединения, перезагрузка).
3. Проверка точности:
   * на тестовой выборке разговоров;
   * сравнение с ручной разметкой.
4. Юзабилити‑тестирование дашборда.

#### Этап 7. Развёртывание и мониторинг
1. Контейнеризация приложения (Docker).
2. Настройка CI/CD‑пайплайна.
3. Реализация мониторинга:
   * метрики производительности;
   * статистика оповещений;
   * логи ошибок.
4. Настройка оповещений о сбоях системы.
5. Документирование системы.

#### Этап 8. Обучение пользователей
1. Подготовка инструкций для:
   * супервизоров (работа с оповещениями);
   * операторов (правила поведения);
   * администраторов (настройка правил).
2. Проведение обучающих сессий.
3. Создание базы знаний и FAQ.

#### Этап 9. Пилотное внедрение
1. Запуск на ограниченном числе операторов (5–10 человек).
2. Сбор обратной связи от супервизоров.
3. Корректировка порогов и правил детекции.
4. Оптимизация производительности.
5. Постепенное масштабирование на весь колл‑центр.
